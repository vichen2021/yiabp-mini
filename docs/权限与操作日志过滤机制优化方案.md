# 权限与操作日志过滤机制优化方案

## 背景

当前权限过滤核心在：

```text
Yi.Abp/framework/operation/Yi.Framework.Operation.Core/Metadata/DefaultActionMetadataResolver.cs
Yi.Abp/framework/operation/Yi.Framework.Operation.Core/Filters/PermissionAuthorizationFilter.cs
Yi.Abp/module/audit-logging/Yi.Module.AuditLogging.Domain/DefaultOperationLogMapper.cs
```

现有实现把权限码推断、操作日志标题推断、模块名推断、实体名推断、操作类型推断都放在同一个 `ActionMetadata` 解析流程中。短期能减少重复代码，但长期会让权限和日志互相牵制。

本项目又不能简单改成所有接口都显式 `[Permission]`，因为标准 CRUD 方法由 `YiCrudAppService` 基类提供，并通过 ABP Conventional Controller 自动暴露为 API。业务服务通常只继承基类，不会为每个 CRUD 方法重写一遍方法。如果为了挂权限特性而强制重写，会破坏 CRUD 基类的价值。

因此优化目标不是取消自动过滤，而是把自动过滤从“根据命名空间和方法名前缀猜测”升级为“资源契约 + 基类 CRUD 显式动作声明 + 自定义方法显式声明”。

## 当前问题

### 1. 权限和日志职责耦合

当前 `ActionMetadata` 同时包含：

```text
PermissionCode
ExplicitPermissionCode
IgnorePermission
OperType
LogTitle
ExplicitLogInfo
IgnoreLog
IsWriteOperation
```

权限是执行前的安全准入，操作日志是执行后的审计投影。两者默认策略不同：

- 权限未知时应逐步走向拒绝。
- 日志未知时可以选择不记录。

把两者绑定在同一个解析结果上，会导致权限规则调整影响日志，日志字段调整影响权限。

### 2. 模块名靠命名空间硬编码

当前类似逻辑不应长期保留：

```csharp
if (ns.Contains("Yi.Module.Rbac")) return "system";
if (ns.Contains("Yi.Module.AuditLogging")) return "monitor";
return "system";
```

问题：

- 新模块会静默落到 `system`。
- 命名空间重构会影响权限码。
- 业务模块概念被写死在框架层。
- 服务类本身看不出自己的权限资源。

### 3. 方法名前缀推断应移除，改为显式动作声明

`Update*` 映射为 `edit`，`Create*` 映射为 `add`，`Delete*` 映射为 `remove` 在语义上成立，但不应该再由 resolver 通过字符串前缀推断。框架基类 CRUD 方法应在 `YiCrudAppService` 上直接标记 `[PermissionAction]`；业务自定义方法也必须显式标记 `[PermissionAction]` 或 `[Permission]`。

例如：

```text
UpdateProfileAsync       必须显式声明 [PermissionAction("edit")] 或 [Permission("...")]
UpdateStateAsync         必须显式声明 [PermissionAction("edit")] / [PermissionAction("update")] / [Permission("...")]
CreateAuthUserAsync      必须显式声明 [PermissionAction("add")] 或 [Permission("role:auth-user:add")]
DeleteAuthUserAsync      必须显式声明 [PermissionAction("remove")] 或 [Permission("role:auth-user:remove")]
```

因此优化重点是：CRUD 的自动能力来自基类方法上的 `[PermissionAction]`，不是来自方法名前缀；自定义方法如果要参与权限过滤，必须显式表达权限动作。

### 4. 未解析默认放行存在安全风险

未解析接口如果直接放行，会让新增接口绕过权限边界。权限解析失败应作为开发期问题暴露出来，并通过资源声明、方法命名或显式权限修正。

### 5. 权限动作词表不统一

现有代码和菜单种子中存在多套动作词：

```text
list / query / detail
delete / remove
edit / update
```

后端自动生成的权限码如果和菜单权限码不一致，会导致用户拥有菜单权限但后端拒绝，或者后端校验了一个菜单系统没有维护的权限码。

## 设计原则

1. 保留自动过滤，服务于基类 CRUD 自动 API。
2. 权限和操作日志拆分决策，不再共用一个“大而全”的 `ActionMetadata`。
3. 共享稳定的 Action 身份解析，不复制服务类型、方法、资源、CRUD 动作识别逻辑。
4. 模块和资源优先来自显式契约和命名规范，自动推断只作为可解释 fallback。
5. 自动动作解析来自基类 CRUD 方法上的 `[PermissionAction]`。
6. 自定义方法必须通过 `[PermissionAction]` 或 `[Permission]` 显式声明。
7. 迁移期兼容旧规则，但新规则默认可测试、可发现、可失败。

## 目标架构

优化后的结构：

```text
ControllerActionDescriptor / ServiceType + MethodInfo
        |
        v
IActionIdentityResolver
        |
        +--> IPermissionRequirementResolver
        |
        +--> IOperationLogRequirementResolver
```

### IActionIdentityResolver

只解析稳定事实，不做权限或日志决策：

```csharp
public sealed class ActionIdentity
{
    public Type ServiceType { get; set; }
    public MethodInfo MethodInfo { get; set; }
    public string? Module { get; set; }
    public string? Resource { get; set; }
    public string? CrudAction { get; set; }
    public bool IsCrudAction { get; set; }
    public string? RemoteServiceName { get; set; }
}
```

它负责：

- 识别服务类型。
- 识别执行方法。
- 读取资源声明。
- 识别 ABP RemoteServiceName。
- 识别是否为已知 CRUD 方法。
- 生成稳定缓存 key。

它不负责：

- 判断是否有权限。
- 判断是否记录日志。
- 生成日志标题。
- 决定未解析时放行或拒绝。

### IPermissionRequirementResolver

输出权限要求：

```csharp
public sealed class PermissionRequirement
{
    public bool Ignore { get; set; }
    public bool IsResolved { get; set; }
    public string? Code { get; set; }
    public string? Source { get; set; }
    public string? UnresolvedReason { get; set; }
}
```

权限解析只关心：

- `[IgnorePermission]`
- `[Permission]`
- `[PermissionAction]`
- 类级资源声明
- CRUD 动作到权限码的映射

### IOperationLogRequirementResolver

输出操作日志要求：

```csharp
public sealed class OperationLogRequirement
{
    public bool Ignore { get; set; }
    public bool ShouldLog { get; set; }
    public OperEnum? OperType { get; set; }
    public string? Title { get; set; }
    public bool SaveRequestData { get; set; }
    public bool SaveResponseData { get; set; }
}
```

日志解析只关心：

- `[IgnoreOperLog]`
- `[OperLog]`
- `[OperLogEntity]`
- `OperationOptions.AutoLogWriteOperations`
- `OperationOptions.LogReadOperations`
- `OperationOptions.IgnoredUrlPrefixes`
- CRUD 动作到日志类型和标题的映射

## 新增资源契约

### PermissionResourceAttribute

用于服务类或接口，声明权限资源：

```csharp
[AttributeUsage(AttributeTargets.Class | AttributeTargets.Interface)]
public sealed class PermissionResourceAttribute : Attribute
{
    public string Module { get; }
    public string Resource { get; }

    public PermissionResourceAttribute(string module, string resource)
    {
        Module = module;
        Resource = resource;
    }
}
```

示例：

```csharp
[PermissionResource("system", "user")]
[OperLogEntity("用户")]
public class UserService : YiCrudAppService<...>
{
}
```

标准 CRUD 自动生成：

```text
GetListAsync              -> system:user:query
GetAsync                  -> system:user:query
CreateAsync               -> system:user:add
UpdateAsync               -> system:user:edit
DeleteAsync(IEnumerable)  -> system:user:remove
GetExportExcelAsync       -> system:user:export
PostImportExcelAsync      -> system:user:import
```

### PermissionActionAttribute

用于基类 CRUD 方法和业务自定义方法，只声明动作段：

```csharp
[AttributeUsage(AttributeTargets.Method)]
public sealed class PermissionActionAttribute : Attribute
{
    public string Action { get; }

    public PermissionActionAttribute(string action)
    {
        Action = action;
    }
}
```

基类 CRUD 方法示例：

```csharp
[PermissionAction("query")]
public override async Task<PagedResultDto<TGetListOutputDto>> GetListAsync(TGetListInput input)

[PermissionAction("add")]
public override async Task<TGetOutputDto> CreateAsync(TCreateInput input)

[PermissionAction("edit")]
public override async Task<TGetOutputDto> UpdateAsync(TKey id, TUpdateInput input)

[PermissionAction("remove")]
public virtual async Task DeleteAsync(IEnumerable<TKey> ids)

[PermissionAction("export")]
public virtual async Task<IActionResult> GetExportExcelAsync(TGetListInput input)

[PermissionAction("import")]
public virtual Task PostImportExcelAsync(List<TCreateInput> input)
```

解析器实现要求：

```text
1. 优先读取当前方法上的 [PermissionAction]。
2. 当前方法没有时，读取 GetBaseDefinition() 对应基类方法上的 [PermissionAction]。
3. 如果方法来自接口代理，还需要查找接口方法上的 [PermissionAction]。
4. 不能再通过方法名前缀生成动作。
```

业务自定义方法示例：

```csharp
[PermissionAction("update")]
[OperLog("更新用户状态", OperEnum.Update)]
public Task<UserGetOutputDto> UpdateStateAsync(Guid id, bool state)
{
}
```

如果服务类有：

```csharp
[PermissionResource("system", "user")]
```

则生成：

```text
system:user:update
```

### PermissionAttribute

保留完整权限码显式声明，用于特殊场景：

```csharp
[Permission("monitor:operlog:remove")]
public Task DeleteCleanAsync()
{
}
```

显式 `[Permission]` 优先级最高，不依赖资源声明。

## 模块前缀策略

模块前缀优先级：

```text
1. [PermissionResource("module", "resource")]
2. ABP Conventional Controller RemoteServiceName
3. 程序集名或模块名规范化 fallback
4. unresolved
```

不再允许无条件默认 `system`。

### RemoteServiceName 对齐

当前 Web 配置中已经指定：

```text
rbac              -> 当前业务期望 system
tenant-management -> 当前业务期望 system
audit             -> 当前业务期望 monitor
default           -> 当前业务期望 system
```

权限模块前缀不写入 `appsettings.json`。如果自动模块名前缀和权限码不一致，优先通过以下方式修正：

```text
1. 调整前后端模块名，使 RemoteServiceName、菜单权限码、后端权限码一致。
2. 在服务类上使用 [PermissionResource("module", "resource")] 显式声明。
3. 对特殊方法使用 [Permission("完整权限码")]。
```

新模块未显式声明时，可以从 RemoteServiceName 生成候选模块名：

```text
audit-logging
tenant-management
setting-management
```

命名格式建议统一为 kebab-case，不建议使用 camelCase：

```text
audit-logging
tenant-management
```

但候选值只能作为 fallback。若菜单权限码使用 `monitor`，就必须显式声明为 `monitor`，或调整前后端模块命名保持一致。

## 动作词表

建议统一采用当前菜单系统更常见的动作：

```text
query
add
edit
remove
export
import
```

兼容期可以保留别名：

```text
list   -> query
detail -> query
delete -> remove
update -> edit 或业务自定义 update
```

内部新生成的权限码应只生成标准动作词，避免后续继续扩散。

### CRUD 方法动作声明

基类 CRUD 方法在 `YiCrudAppService` 中显式声明 `[PermissionAction]`：

```text
GetListAsync              [PermissionAction("query")]
GetSelectDataListAsync    [PermissionAction("query")]
GetAsync                  [PermissionAction("query")]
CreateAsync               [PermissionAction("add")]
UpdateAsync               [PermissionAction("edit")]
DeleteAsync(TKey)         [PermissionAction("remove")]
DeleteAsync(IEnumerable)  [PermissionAction("remove")]
GetExportExcelAsync       [PermissionAction("export")]
PostImportExcelAsync      [PermissionAction("import")]
```

说明：

- `DeleteAsync(TKey)` 如果远程服务禁用，则不会进入公开 API 校验。
- `GetAsync` 和 `GetListAsync` 是否共用 `query` 要与菜单种子保持一致。
- `GetSelectDataListAsync` 如果只服务下拉框，可以通过 `[IgnorePermission]` 忽略，或在基类上声明为 `query`。

### 非 CRUD 方法

以下业务自定义方法不再按前缀自动推断，必须显式声明：

```text
UpdateStateAsync
UpdatePasswordAsync
UpdateProfileAsync
CreateAuthUserAsync
DeleteAuthUserAsync
DeleteCleanAsync
PostExportAsync
```

处理方式：

```text
1. [Permission("完整权限码")] 声明完整权限
2. [PermissionAction("动作")] 声明动作段，并结合 [PermissionResource] 生成权限码
3. [IgnorePermission] 明确忽略
```

## 权限过滤流程

授权判断顺序：

```text
1. [AllowAnonymous] -> 放行
2. [IgnorePermission] -> 放行
3. 白名单 -> 放行
4. [Permission("...")] -> 校验完整权限码
5. [PermissionAction("...")] + 资源声明 -> 校验组合权限码
6. 未解析 -> 403，并输出诊断信息
```

显式 `[Permission]` 必须校验，即使无法解析模块、资源或 CRUD 动作。

## 操作日志流程

操作日志不再使用权限的 `PermissionRequirement`。它基于 `ActionIdentity` 自己生成 `OperationLogRequirement`。

判断顺序：

```text
1. URL 命中 IgnoredUrlPrefixes -> 不记录
2. [IgnoreOperLog] -> 不记录
3. [OperLog] -> 使用显式标题和 OperType
4. CRUD 写操作 + AutoLogWriteOperations = true -> 自动记录
5. CRUD 查询操作 + LogReadOperations = true -> 自动记录
6. 其他未识别方法 -> 不自动记录，除非有 [OperLog]
```

日志动作映射：

```text
add     -> OperEnum.Insert -> 添加{实体显示名}
edit    -> OperEnum.Update -> 更新{实体显示名}
remove  -> OperEnum.Delete -> 删除{实体显示名}
export  -> OperEnum.Export -> 导出{实体显示名}
import  -> OperEnum.Import -> 导入{实体显示名}
```

特殊动作使用 `[OperLog]`：

```csharp
[OperLog("清空操作日志", OperEnum.Clear)]
[Permission("monitor:operlog:remove")]
public Task DeleteCleanAsync()
{
}
```

## 配置边界

权限相关规则不写入 `appsettings.json`：

```text
不写入权限模块映射。
不写入权限码映射项。
不写入未解析权限放行开关。
不写入 legacy 命名空间推断开关。
不写入自动权限校验开关。
```

权限行为通过代码契约表达：

```text
[PermissionResource]
[PermissionAction]
[Permission]
[IgnorePermission]
RemoteServiceName / 模块命名规范
基类 CRUD 上的 [PermissionAction]
自定义方法上的 [PermissionAction] 或 [Permission]
```

如果自动生成的模块名不符合预期，优先调整前后端模块名和菜单权限码，使命名体系一致；特殊情况使用特性覆盖。

操作日志属于运行策略，可以继续使用 `OperationOptions` 控制是否记录、是否记录读操作、忽略哪些 URL。

## 需要删除或弱化的旧逻辑

最终应删除：

```text
DefaultActionMetadataResolver.InferModuleName 中的命名空间硬编码
DefaultActionMetadataResolver.InferByPrefix 方法名前缀推断
权限和日志共用的大 ActionMetadata 决策模型
无条件 return "system" fallback
```

可短期保留但必须标记为 legacy：

```text
LegacyNamespaceInference
```

该兼容逻辑只用于迁移期，不暴露到 `appsettings.json`。迁移完成后删除。

## 迁移计划

### Phase 1：补充抽象，不改变行为

1. 新增 `PermissionResourceAttribute`。
2. 新增 `PermissionActionAttribute`。
3. 新增 `ActionIdentity`。
4. 新增 `IActionIdentityResolver`。
5. 现有 resolver 暂时可以继续工作。

验收：

```text
构建通过。
现有权限行为不变。
```

### Phase 2：拆分权限和日志决策

1. 新增 `IPermissionRequirementResolver`。
2. 新增 `IOperationLogRequirementResolver`。
3. `PermissionAuthorizationFilter` 改用 `IPermissionRequirementResolver`。
4. `DefaultOperationLogMapper` 改用 `IOperationLogRequirementResolver`。
5. 两者共享 `IActionIdentityResolver`。

验收：

```text
权限修改不影响日志。
日志标题修改不影响权限。
```

### Phase 3：接入资源声明

给核心服务补充：

```text
UserService              -> [PermissionResource("system", "user")]
RoleService              -> [PermissionResource("system", "role")]
MenuService              -> [PermissionResource("system", "menu")]
DeptService              -> [PermissionResource("system", "dept")]
PostService              -> [PermissionResource("system", "post")]
DictionaryService        -> [PermissionResource("system", "dict")]
DictionaryTypeService    -> [PermissionResource("system", "dict-type")]
ConfigService            -> [PermissionResource("system", "config")]
NoticeService            -> [PermissionResource("system", "notice")]
TenantService            -> [PermissionResource("system", "tenant")]
TenantPackageService     -> [PermissionResource("system", "tenant-package")]
OperationLogService      -> [PermissionResource("monitor", "operlog")]
LoginLogService          -> [PermissionResource("monitor", "loginlog")]
```

具体资源名必须以菜单种子里的权限码为准。

### Phase 4：移除方法名前缀推断

1. 在 `YiCrudAppService` 基类 CRUD 方法上补齐 `[PermissionAction]`。
2. Resolver 读取当前方法、基类方法和接口方法上的 `[PermissionAction]`。
3. 删除 `InferByPrefix`。
4. 业务自定义方法批量扫描并补充 `[PermissionAction]`、`[Permission]` 或 `[IgnorePermission]`。
5. 输出未解析接口诊断日志。

验收：

```text
UpdateProfileAsync 未显式声明时不生成权限码，并在严格模式下返回 403。
CreateAuthUserAsync 未显式声明时不生成权限码，并在严格模式下返回 403。
```

### Phase 5：关闭 legacy fallback

1. 未解析权限固定返回 403。
2. 删除 legacy namespace inference。
3. 删除 `InferModuleName` 命名空间硬编码。
4. 清理旧兼容代码。

验收：

```text
未声明资源的新 API 默认不会静默获得 system 权限码。
新增模块不需要修改框架代码。
```

## 测试与验收

### 权限验收

- `[AllowAnonymous]` 放行。
- `[IgnorePermission]` 放行。
- 显式 `[Permission]` 必须校验。
- `[PermissionAction]` 能结合类级资源生成权限码。
- 基类 CRUD 方法通过基类 `[PermissionAction]` 自动生成正确权限码。
- 业务自定义方法必须显式声明 `[PermissionAction]`、`[Permission]` 或 `[IgnorePermission]`。
- 未解析方法返回 403，并输出诊断信息。
- 无权限返回 403。
- 有权限正常执行业务。

### 操作日志验收

- `[IgnoreOperLog]` 不记录。
- `[OperLog]` 使用显式标题和类型。
- CRUD 写操作自动记录。
- 查询操作默认不记录。
- `AutoLogWriteOperations = false` 时，普通写操作不自动记录。
- 显式 `[OperLog]` 不受 `AutoLogWriteOperations` 影响。
- `IgnoredUrlPrefixes` 命中时不记录。

### 模块边界验收

- `Operation.Core` 不依赖 RBAC 或 AuditLogging 持久化。
- `AuditLogging.Domain` 不依赖 RBAC。
- 权限过滤不依赖操作日志实体。
- 操作日志映射不依赖权限校验结果。
- 两者只共享 `ActionIdentity`。

最终结构应收敛为：

```text
Operation.Abstractions
  PermissionAttribute
  PermissionActionAttribute
  PermissionResourceAttribute
  IgnorePermissionAttribute
  OperLogAttribute
  OperLogEntityAttribute
  IgnoreOperLogAttribute
  ActionIdentity
  PermissionRequirement
  OperationLogRequirement
  IActionIdentityResolver
  IPermissionRequirementResolver
  IOperationLogRequirementResolver

Operation.Core
  DefaultActionIdentityResolver
  DefaultPermissionRequirementResolver
  DefaultOperationLogRequirementResolver
  PermissionAuthorizationFilter
  DefaultPermissionHandler

AuditLogging.Domain
  DefaultOperationLogMapper
  OperationLogEntity
  OperationLogRepository abstraction

Rbac
  菜单、角色、用户、权限码数据
```

一句话总结：

```text
自动权限仍然保留，但只对 CRUD 约定自动；权限和日志各自决策；模块和资源来自显式契约；命名空间硬编码最终删除。
```
