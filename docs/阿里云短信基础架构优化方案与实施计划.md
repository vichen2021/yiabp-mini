# 阿里云短信基础架构优化方案与实施计划

## 背景

当前项目已经存在阿里云短信的基础接入代码，主要包括：

- `AliyunOptions`
- `AliyunManger`
- `IAliyunManger`
- `AccountService` 中的手机验证码发送调用
- `AlibabaCloud.SDK.Dysmsapi20170525` SDK 依赖
- `YiModuleRbacDomainModule` 中的 Options 绑定

当前实现可以视为“基础骨架已存在”，但还不适合作为完整基础设施能力交付。

主要问题：

- `appsettings.json` 缺少短信配置示例。
- `AliyunOptions.Sms` 可能为空，存在空引用风险。
- 发送前没有配置完整性校验。
- 发送后没有校验阿里云业务返回码。
- `AliyunManger` 命名存在拼写问题。
- `AliyunManger` 命名过于宽泛，容易与阿里云 OSS 等其他阿里云能力混淆。

---

## 命名结论

不建议简单改为：

```csharp
AliyunManager
IAliyunManager
```

原因是项目即将上线阿里云 OSS 文件存储能力，未来还可能接入更多阿里云服务，例如：

- OSS
- SMS
- CDN
- STS
- DM

如果使用 `AliyunManager`，语义过宽，会让人误以为它管理所有阿里云能力。

更推荐改为：

```csharp
AliyunSmsManager
IAliyunSmsManager
```

该命名明确表示这是阿里云短信能力，不会与 OSS 配置、OSS Provider 或其他阿里云服务冲突。

如果后续希望进一步解耦供应商，则可再增加通用接口：

```csharp
ISmsSender
```

并由 `AliyunSmsManager` 或 `AliyunSmsSender` 实现。

---

## 推荐命名策略

### 当前阶段推荐

当前阶段建议采用：

```csharp
IAliyunSmsManager
AliyunSmsManager
```

适合原因：

- 改动范围较小。
- 和现有 `AliyunManger` 迁移路径清晰。
- 明确区分阿里云短信和阿里云 OSS。
- 不需要立刻设计多短信供应商体系。

### 后续扩展推荐

如果未来要支持多个短信供应商，例如阿里云、腾讯云、华为云、控制台模拟短信，则建议引入：

```csharp
ISmsSender
AliyunSmsSender
TencentSmsSender
ConsoleSmsSender
```

业务服务依赖：

```csharp
ISmsSender
```

而不是直接依赖：

```csharp
IAliyunSmsManager
```

---

## 目标

本次优化目标：

- 修复 `AliyunManger` 拼写问题。
- 将命名升级为 `AliyunSmsManager`，避免与 OSS 冲突。
- 补齐短信配置示例。
- 避免 Options 空引用。
- 发送前校验必要配置。
- 发送后校验阿里云返回结果。
- 保持现有业务调用行为不变。
- 不引入复杂的多供应商短信架构。

---

## 非目标

- 不实现短信供应商运行时切换。
- 不实现腾讯云、华为云短信。
- 不实现短信发送记录表。
- 不实现短信频控的完整风控系统。
- 不修改手机验证码业务流程。
- 不把短信能力与阿里云 OSS 配置合并为同一个 Options。

---

## 当前实现分析

### Options

当前文件：

```text
Yi.Abp/module/rbac/Yi.Module.Rbac.Domain.Shared/Options/AliyunOptions.cs
```

当前结构：

```csharp
public class AliyunOptions
{
    public string AccessKeyId { get; set; }
    public string AccessKeySecret { get; set; }
    public AliyunSms Sms { get; set; }
}

public class AliyunSms
{
    public string SignName { get; set; }
    public string TemplateCode { get; set; }
}
```

存在问题：

- 字符串属性未初始化。
- `Sms` 未初始化。
- 配置缺失时可能出现空引用。

---

### Manager

当前文件：

```text
Yi.Abp/module/rbac/Yi.Module.Rbac.Domain/Managers/AliyunManger.cs
```

当前接口：

```csharp
public interface IAliyunManger
{
    Task SendSmsAsync(string phoneNumbers, string code);
}
```

存在问题：

- `Manger` 拼写错误。
- 名称过宽，容易和 OSS 混淆。
- 只捕获异常，不校验返回码。
- 没有配置校验。
- 发送失败时日志信息不够结构化。

---

"BlobStoring": {

```json
"AliyunOptions": {

    "SignName": "",
    "TemplateCode": ""
  }

```

## 设计方案

### 一、Options 默认值初始化

建议将 `AliyunOptions` 改为：

```csharp
public class AliyunOptions
{
    public string AccessKeyId { get; set; } = string.Empty;
    public string AccessKeySecret { get; set; } = string.Empty;
    public AliyunSmsOptions Sms { get; set; } = new();
}

public class AliyunSmsOptions
{
    public string SignName { get; set; } = string.Empty;
    public string TemplateCode { get; set; } = string.Empty;
}
```

是否保留 `AliyunSms` 名称：

- 如果希望最小改动，可以保留 `AliyunSms`。
- 如果希望语义更规范，建议改为 `AliyunSmsOptions`。

推荐：改为 `AliyunSmsOptions`。

---

### 二、Manager 重命名

建议从：

```text
AliyunManger.cs
AliyunManger
IAliyunManger
```

改为：

```text
AliyunSmsManager.cs
AliyunSmsManager
IAliyunSmsManager
```

业务调用从：

```csharp
private readonly IAliyunManger _aliyunManger;
```

改为：

```csharp
private readonly IAliyunSmsManager _aliyunSmsManager;
```

调用从：

```csharp
await _aliyunManger.SendSmsAsync(input.Phone, code);
```

改为：

```csharp
await _aliyunSmsManager.SendSmsAsync(input.Phone, code);
```

---

### 三、配置校验

在 `SendSmsAsync` 内发送前校验：

- `AccessKeyId`
- `AccessKeySecret`
- `Sms.SignName`
- `Sms.TemplateCode`

缺失时抛出明确异常：

```text
阿里云短信配置不完整，请检查 AliyunOptions 配置节点
```

建议不要把真实配置值打印到日志，避免泄露密钥。

---

### 四、发送结果校验

阿里云短信 SDK 调用成功不代表短信业务成功。

应检查响应体中的业务状态码：

```csharp
var response = await client.SendSmsAsync(sendSmsRequest);

if (!string.Equals(response.Body.Code, "OK", StringComparison.OrdinalIgnoreCase))
{
    _logger.LogWarning(
        "阿里云短信发送失败，Code:{Code}, Message:{Message}, RequestId:{RequestId}",
        response.Body.Code,
        response.Body.Message,
        response.Body.RequestId);

    throw new UserFriendlyException($"短信发送失败：{response.Body.Message}");
}
```

这样可以准确区分：

- HTTP 调用异常。
- 阿里云业务返回失败。
- 配置缺失。

---

### 五、补充 appsettings 示例配置

建议在 `src/Yi.Abp.Web/appsettings.json` 增加：

```json
"AliyunOptions": {
  "AccessKeyId": "",
  "AccessKeySecret": "",
  "Sms": {
    "SignName": "",
    "TemplateCode": ""
  }
}
```

注意：

- 开源仓库只提交空字符串或占位符。
- 不提交真实 AccessKey。
- 生产环境通过环境变量、用户机密或私有配置文件覆盖。

---

## 与阿里云 OSS 功能的关系

阿里云 OSS 当前建议保留在：

```json
"BlobStoring": {
  "Provider": "FileSystem",
  "Aliyun": {}
}
```

阿里云短信建议继续使用：

```json
"AliyunOptions": {
  "Sms": {}
}
```

短期不建议强行合并为：

```json
"Aliyun": {
  "Sms": {},
  "Oss": {}
}
```

原因：

- OSS 当前绑定的是 ABP BlobStoring Provider 配置模型。
- 短信是自定义基础设施配置。
- 强行合并会扩大改造范围。
- 当前目标是稳定上线 OSS，同时补齐短信基础能力。

命名层面应通过 `AliyunSmsManager` 明确区分短信能力，避免 `AliyunManager` 和 OSS 产生概念冲突。

## 实施计划

### 阶段一：配置与 Options 加固

1. 修改 `AliyunOptions`，增加默认值初始化。
2. 将 `AliyunSms` 改名为 `AliyunSmsOptions`。
3. 在 `appsettings.json` 增加 `AliyunOptions` 示例节点。
4. 构建 `Yi.Module.Rbac.Domain.Shared`。

验收标准：

- 缺少短信配置时不再出现 `NullReferenceException`。
- 配置结构清晰可见。
- 开源配置不包含真实密钥。

---

### 阶段二：Manager 重命名

1. 将文件 `AliyunManger.cs` 改为 `AliyunSmsManager.cs`。
2. 将类名 `AliyunManger` 改为 `AliyunSmsManager`。
3. 将接口 `IAliyunManger` 改为 `IAliyunSmsManager`。
4. 更新 `AccountService` 构造函数依赖。
5. 更新字段名 `_aliyunManger` 为 `_aliyunSmsManager`。
6. 全局搜索确认无 `AliyunManger` 残留。

验收标准：

- 项目编译通过。
- DI 可以正常解析 `IAliyunSmsManager`。
- `AccountService` 手机验证码发送调用不变。

---

### 阶段三：发送前配置校验

1. 在 `AliyunSmsManager` 中增加配置校验方法。
2. 发送前检查必要配置。
3. 缺失配置时抛出明确的 `UserFriendlyException`。
4. 日志中不输出 AccessKeySecret。

验收标准：

- 配置缺失时返回明确错误。
- 不再由 SDK 或空引用异常暴露底层错误。
- 日志不泄露敏感信息。

---

### 阶段四：发送结果校验

1. 获取 `SendSmsAsync` 响应结果。
2. 判断 `response.Body.Code` 是否为 `OK`。
3. 非 `OK` 时记录 `Code`、`Message`、`RequestId`。
4. 抛出用户友好异常。

验收标准：

- 阿里云业务失败能被准确识别。
- 日志能辅助定位失败原因。
- 前端不会误认为发送成功。

---

### 阶段五：构建与手工验证

建议验证顺序：

1. 构建 `Yi.Module.Rbac.Domain.Shared`。
2. 构建 `Yi.Module.Rbac.Domain`。
3. 构建 `Yi.Module.Rbac.Application`。
4. 使用空配置调用短信接口，确认返回配置缺失错误。
5. 使用真实测试签名和模板调用短信接口，确认发送成功。
6. 使用错误模板 Code 调用短信接口，确认能识别业务失败。

---

## 后续演进方向

如果后续要把短信能力做成真正通用基础设施，建议演进为：

```csharp
public interface ISmsSender
{
    Task SendAsync(string phoneNumber, string templateCode, object templateParams);
}
```

然后提供：

```csharp
AliyunSmsSender
ConsoleSmsSender
```

业务层依赖 `ISmsSender`，配置决定使用真实短信还是本地日志短信。

但这属于下一阶段，不建议和当前 OSS 上线、短信修复一起做，避免扩大范围。

---

## 结论

`AliyunManger` 不建议改为 `AliyunManager`。

更推荐改为：

```csharp
AliyunSmsManager
IAliyunSmsManager
```

理由：

- 修复拼写问题。
- 明确短信职责。
- 避免和阿里云 OSS 功能混淆。
- 改动范围可控。
- 为未来抽象 `ISmsSender` 留出空间。

本次建议采用“小步加固”方案：

- 先完成短信 Options、配置、命名、结果校验。
- 不和 OSS 配置模型强行合并。
- 不立即引入多短信供应商抽象。
