# 权限动作枚举化设计方案与实施计划

## 背景

当前权限动作通过字符串声明，例如：

```csharp
[PermissionAction("query")]
[PermissionAction("add")]
[PermissionAction("edit")]
[PermissionAction("remove")]
```

这类字符串分散在 `YiCrudAppService`、`MenuService`、`FileService`、`MonitorCacheService` 等应用服务中。随着模块增多，字符串方式容易出现拼写不一致、大小写不统一、重构困难等问题。

项目当前已经基本形成统一动作语义：

- `query`
- `add`
- `edit`
- `remove`
- `export`
- `import`

因此适合将权限动作声明从裸字符串升级为枚举，以提升类型安全和可维护性。

---

## 目标

- 将常用权限动作统一定义为枚举。
- 支持使用枚举声明权限动作，例如：

```csharp
[PermissionAction(PermissionActionEnum.Query)]
```

- 保持原有字符串构造方式兼容，避免一次性破坏历史代码。
- 权限码最终仍然保持小写字符串格式，例如：

```text
system:menu:query
system:menu:add
system:menu:edit
system:menu:remove
```

- 不改变现有权限码格式、鉴权流程、菜单权限码结构和数据库字段。

---

## 非目标

- 不修改 `PermissionResource` 的 `module/resource` 字符串声明方式。
- 不修改 `PermissionAttribute("module:resource:action")` 显式权限码能力。
- 不引入多语言权限动作名称。
- 不改变菜单表 `PermissionCode` 字段结构。
- 不改变 `DefaultPermissionHandler` 的权限匹配和通配符逻辑。

---

## 是否推荐枚举化

推荐。

原因如下：

### 1. 避免拼写错误

字符串写法无法在编译期检查：

```csharp
[PermissionAction("qurey")]
```

这种错误只有运行时权限解析或接口访问时才会暴露。

枚举写法可以在编译期约束：

```csharp
[PermissionAction(PermissionActionEnum.Query)]
```

### 2. 有利于全局统一

当前菜单种子、CRUD 基类、业务服务都已经统一到 `query/add/edit/remove` 这套动作。枚举可以作为唯一动作字典，减少模块各自定义动作的概率。

### 3. 重构更安全

如果后续要调整动作命名或补充动作，可以通过枚举和映射集中管理。

### 4. 保留兼容性后迁移成本低

`PermissionActionAttribute` 可以同时保留字符串构造函数和新增枚举构造函数，做到渐进迁移。

---

## 设计方案

### 一、定义权限动作枚举

建议放在授权抽象层：

```text
Yi.Abp/framework/authorization/Yi.Framework.Authorization.Abstractions/Enums/PermissionActionEnum.cs
```

建议枚举：

```csharp
namespace Yi.Framework.Authorization.Abstractions.Enums;

public enum PermissionActionEnum
{
    Query,
    Add,
    Edit,
    Remove,
    Export,
    Import
}
```

命名建议使用 `PermissionActionEnum`，避免与现有 `PermissionActionAttribute` 概念混淆。

---

### 二、增加枚举到权限码字符串的转换

由于权限码第三段要求小写字符串，枚举不能直接使用 `ToString()`，否则会得到 `Query`、`Add` 这类 PascalCase 字符串。

建议新增扩展方法：

```text
Yi.Abp/framework/authorization/Yi.Framework.Authorization.Abstractions/Extensions/PermissionActionEnumExtensions.cs
```

设计：

```csharp
namespace Yi.Framework.Authorization.Abstractions.Extensions;

public static class PermissionActionEnumExtensions
{
    public static string ToPermissionCode(this PermissionActionEnum action)
    {
        return action switch
        {
            PermissionActionEnum.Query => "query",
            PermissionActionEnum.Add => "add",
            PermissionActionEnum.Edit => "edit",
            PermissionActionEnum.Remove => "remove",
            PermissionActionEnum.Export => "export",
            PermissionActionEnum.Import => "import",
            _ => throw new ArgumentOutOfRangeException(nameof(action), action, null)
        };
    }
}
```

这样可以保证最终权限码仍然稳定输出小写动作名。

---

### 三、扩展 `PermissionActionAttribute`

当前属性定义：

```csharp
public sealed class PermissionActionAttribute : Attribute
{
    public string Action { get; }

    public PermissionActionAttribute(string action)
    {
        Action = action;
    }
}
```

建议改造为兼容写法：

```csharp
public sealed class PermissionActionAttribute : Attribute
{
    public string Action { get; }

    public PermissionActionAttribute(string action)
    {
        Action = action;
    }

    public PermissionActionAttribute(PermissionActionEnum action)
    {
        Action = action.ToPermissionCode();
    }
}
```

这样原有代码继续可用，新代码可以使用枚举。

---

### 四、迁移典型使用点

#### `YiCrudAppService`

从：

```csharp
[PermissionAction("query")]
public override Task<TGetOutputDto> GetAsync(TKey id)
```

迁移为：

```csharp
[PermissionAction(PermissionActionEnum.Query)]
public override Task<TGetOutputDto> GetAsync(TKey id)
```

建议迁移动作：

- `query` -> `PermissionActionEnum.Query`
- `add` -> `PermissionActionEnum.Add`
- `edit` -> `PermissionActionEnum.Edit`
- `remove` -> `PermissionActionEnum.Remove`
- `export` -> `PermissionActionEnum.Export`
- `import` -> `PermissionActionEnum.Import`

#### `MenuService`

从：

```csharp
[PermissionAction("query")]
public async Task<List<MenuGetListOutputDto>> GetListAsync(MenuGetListInputVo input)
```

迁移为：

```csharp
[PermissionAction(PermissionActionEnum.Query)]
public async Task<List<MenuGetListOutputDto>> GetListAsync(MenuGetListInputVo input)
```

## 兼容性设计

### 保留字符串构造函数

短期内不删除：

```csharp
public PermissionActionAttribute(string action)
```

原因：

- 项目中可能存在自定义动作。
- 历史模块可以逐步迁移。
- 避免一次性大范围修改导致风险过高。

### 是否需要标记字符串构造函数过时

第一阶段不建议立即加 `[Obsolete]`。

建议在项目完成主要迁移并稳定后，再考虑：

```csharp
[Obsolete("请优先使用 PermissionActionEnum 声明标准权限动作。自定义动作仍可使用字符串。")]
```

但如果项目需要长期支持扩展动作，字符串构造函数可以一直保留，不必标记过时。

---

## 对现有权限解析流程的影响

权限解析器当前读取：

```csharp
var actionAttr = ActionReflectionHelper.GetMethodAttribute<PermissionActionAttribute>(methodInfo, identity.ServiceType);
requirement.Code = $"{resourceAttr.Module}:{resourceAttr.Resource}:{actionAttr.Action}";
```

枚举化后 `Action` 仍然是字符串，因此：

- `DefaultPermissionRequirementResolver` 不需要修改。
- `DefaultPermissionDefinitionProvider` 不需要修改。
- `DefaultPermissionHandler` 不需要修改。
- JWT 权限声明不需要修改。
- 菜单权限码不需要修改结构。

---

## 风险点

### 1. 枚举名称和权限码字符串大小写不一致

必须通过扩展方法显式映射，不能直接使用 `action.ToString()`。

### 2. 自定义动作扩展能力

如果未来某些模块需要标准动作之外的动作，例如：

```text
approve
reject
publish
unpublish
```

有两种处理方式：

- 继续使用字符串构造函数。
- 确认动作通用后再加入枚举。

建议原则：

- 通用动作进入枚举。
- 业务强相关动作先保留字符串。

### 3. 与菜单种子权限码保持一致

枚举只影响服务方法声明。菜单种子仍然存储最终权限码字符串，因此迁移后仍需保证：

```text
module:resource:query
module:resource:add
module:resource:edit
module:resource:remove
```

与服务方法解析结果一致。

## 实施计划

### 阶段一：增加枚举能力

1. 新增 `PermissionActionEnum`。
2. 新增 `PermissionActionEnumExtensions.ToPermissionCode()`。
3. 扩展 `PermissionActionAttribute`，新增枚举构造函数。
4. 构建 `Yi.Framework.Authorization.Abstractions` 项目。

验收标准：

- 原字符串写法仍可编译。
- 新枚举写法可编译。
- `PermissionActionEnum.Query` 输出 `query`。

---

### 阶段二：迁移框架基类

优先迁移：

```text
Yi.Abp/framework/ddd/Yi.Framework.Ddd.Application/YiCrudAppService.cs
```

替换：

```text
"query"  -> PermissionActionEnum.Query
"add"    -> PermissionActionEnum.Add
"edit"   -> PermissionActionEnum.Edit
"remove" -> PermissionActionEnum.Remove
"export" -> PermissionActionEnum.Export
"import" -> PermissionActionEnum.Import
```

验收标准：

- CRUD 基类编译通过。
- 派生服务仍可通过基类方法继承权限动作。
- 分页接口仍解析为 `*:query`。

---

### 阶段三：迁移核心服务

优先迁移以下服务：

```text
Yi.Abp/module/rbac/Yi.Module.Rbac.Application/Services/System/MenuService.cs
Yi.Abp/module/rbac/Yi.Module.Rbac.Application/Services/System/FileService.cs
Yi.Abp/module/rbac/Yi.Module.Rbac.Application/Services/Monitor/MonitorCacheService.cs
```

验收标准：

- `MenuService` 的列表、树、租户套餐菜单树接口仍解析为 `system:menu:query`。
- 文件上传仍解析为 `system:file:add`。
- 文件下载仍解析为 `system:file:query`。
- 缓存删除仍解析为 `monitor:cache:remove` 或对应资源权限码。

---

### 阶段四：全项目扫描迁移

搜索范围：

```text
[PermissionAction("query")]
[PermissionAction("add")]
[PermissionAction("edit")]
[PermissionAction("remove")]
[PermissionAction("export")]
[PermissionAction("import")]
```

逐步替换为枚举写法。

验收标准：

- 标准动作不再使用裸字符串。
- 非标准动作如存在，保留字符串并单独评估是否进入枚举。

---

### 阶段五：构建与接口验证

建议验证：

1. 构建授权抽象层。
2. 构建 DDD Application 层。
3. 构建 RBAC Application 层。
4. 启动 Web 项目。
5. 使用超管账号验证：
   - 菜单列表。
   - 菜单树。
   - 常规 CRUD 分页。
   - 文件上传下载。

---

## 推荐最终代码风格

标准动作使用枚举：

```csharp
[PermissionAction(PermissionActionEnum.Query)]
```

少量业务专用动作允许继续使用字符串：

```csharp
[PermissionAction("approve")]
```

如果某个业务动作多模块复用，再考虑加入 `PermissionActionEnum`。

---

## 结论

建议进行枚举化改造。

推荐采用“兼容式渐进迁移”方案：

- 新增枚举构造函数。
- 保留字符串构造函数。
- 先迁移框架基类和核心服务。
- 再逐步扫描迁移全项目标准动作。

这样可以在不破坏现有权限体系的前提下，提高权限动作声明的类型安全、统一性和可维护性。
