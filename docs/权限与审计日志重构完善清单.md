# 权限与审计日志重构完善清单

## 目标

本清单用于收敛 `Yi.Framework.Operation.*` 与 `audit-logging` 重构后的剩余工作。

最终状态固定为：

- `Operation` 负责业务操作元数据、权限准入、权限码生成。
- `AuditLogging` 负责复用 ABP Auditing 生成 `OperationLog`，并提供操作日志查询、删除、清空、导出。
- `RBAC` 只负责权限业务数据、菜单、角色、用户和权限声明来源。
- 自动生成接口也能稳定获得权限校验和业务操作日志名称。
- 权限和操作日志使用同一套 Action 元数据，避免两套推断规则长期分叉。

## 必须修复项

### 1. 显式 Permission 必须优先校验

当前风险：

如果自定义方法不在自动推断规则内，但方法上存在 `[Permission("...")]`，当前授权过滤器仍可能进入“未解析方法”分支，导致显式权限没有被校验。

固定规则：

```text
1. [AllowAnonymous] -> 放行
2. [IgnorePermission] -> 放行
3. 白名单 -> 放行
4. [Permission("...")] -> 必须校验指定权限码
5. 自动推断出权限码 -> 按 AutoCheckResolvedActions 决定是否校验
6. 未推断且无显式权限：
   - AllowUnresolvedActions = true  -> 放行
   - AllowUnresolvedActions = false -> 403
```

实现要求：

- `metadata.ExplicitPermissionCode` 不为空时，不受 `metadata.IsResolved` 影响。
- `AutoCheckResolvedActions` 只控制自动推断权限，不控制显式 `[Permission]`。
- 多个 `[Permission]` 的处理要确定：当前只取第一个，后续保持“任一满足即通过”或“全部满足才通过”必须明确。当前项目按单权限码模型处理，先保持只支持一个，后续再扩展。

验收：

- 未知方法加 `[Permission("system:user:edit")]` 后必须校验该权限。
- 未知方法未加 `[Permission]` 时才进入 `AllowUnresolvedActions` 分支。

### 2. 权限和操作日志必须共用 ActionMetadata

当前状态：

- 权限使用 `DefaultActionMetadataResolver`。
- 操作日志使用 `DefaultOperationLogMapper` 内部反射和方法名推断。

这会导致权限和日志规则逐渐分叉。

固定目标：

```text
ABP AuditLogInfo.Action
  -> 解析 ServiceType + MethodInfo
  -> 构造统一 ActionMetadata
  -> OperationLogMapper 使用 ActionMetadata 生成标题和操作类型
```

落地方式：

- 保留 `IActionMetadataResolver` 作为 MVC 权限过滤器入口。
- 新增一个面向 `AuditLogActionInfo` 的解析能力，避免 AuditLogging 直接依赖 MVC 的 `ControllerActionDescriptor`。
- 可选实现：

```text
IActionMetadataResolver
  Resolve(ControllerActionDescriptor descriptor)
  Resolve(Type serviceType, MethodInfo methodInfo)
```

或新增：

```text
IActionMetadataProvider
  ActionMetadata Resolve(Type serviceType, MethodInfo methodInfo)
```

确定采用方式：

```text
扩展 IActionMetadataResolver，增加 Resolve(Type serviceType, MethodInfo methodInfo)。
```

原因：

- 继续保持“一个解析器”。
- 权限和操作日志都走同一套实体名、操作名、日志标题、操作类型规则。
- `AuditLogging.Domain` 已依赖 `Operation.Abstractions`，可以注入抽象接口，不引入 RBAC 依赖。

验收：

- `[OperLogEntity("部门")] + UpdateAsync` 在权限和日志元数据里都得到同一个实体显示名。
- `[OperLog("更新部门状态", OperEnum.Update)]` 在操作日志里直接使用特性标题。
- 修改方法名映射时，只需要改一个元数据解析器。

### 3. 完整实现写操作推断规则

文档固定规则：

默认记录写操作：

```text
CreateAsync
UpdateAsync
DeleteAsync
ImportAsync
ExportAsync
Post*
Put*
Delete*
```

默认不记录查询：

```text
GetAsync
GetListAsync
GetSelectDataListAsync
```

当前缺口：

- `Post*`
- `Put*`
- 部分 `Delete*`
- 自定义 `UpdateStateAsync` 需要靠 `[OperLog]` 覆盖。

实现要求：

- `InferActionName` 和 `InferOperType` 使用同一套规则。
- `PostImportExcelAsync` 解析为 `import`。
- `GetExportExcelAsync` 解析为 `export`。
- 普通 `Post*` 默认可解析为 `add` 或 `other`。当前项目采用：

```text
Post*   -> add
Put*    -> edit
Delete* -> delete
```

- 无法判断业务含义的方法不强行生成权限码；特殊动作用 `[Permission]` 和 `[OperLog]` 显式声明。

验收：

- `PostImportExcelAsync` 权限码为 `{module}:{entity}:import`，操作日志为 `导入{实体}`。
- `GetExportExcelAsync` 权限码为 `{module}:{entity}:export`，操作日志为 `导出{实体}`。
- `PostSomethingAsync` 能按规则生成操作日志，或通过 `[OperLog]` 覆盖为准确名称。

### 4. 接入权限码配置映射表

文档要求权限推断优先级：

```text
1. 显式 PermissionAttribute
2. 配置映射表
3. ABP Conventional Controller RemoteServiceName
4. ApplicationService 命名空间和类名
5. 方法名约定
```

当前缺口：

- `PermissionOptions` 没有配置映射表。

固定配置结构：

```json
{
  "Operation": {
    "Permission": {
      "AllowUnresolvedActions": true,
      "Mappings": {
        "Yi.Framework.Rbac.Application.Services.System.UserService.UpdateStateAsync": "system:user:update",
        "Yi.Framework.AuditLogging.Application.Services.OperationLogService.GetListAsync": "monitor:operlog:list"
      }
    }
  }
}
```

实现要求：

- `PermissionOptions` 增加：

```csharp
public Dictionary<string, string> Mappings { get; set; } = new();
```

- 映射 key 固定为：

```text
{FullTypeName}.{MethodName}
```

- 映射优先级低于 `[Permission]`，高于自动推断。

验收：

- 配置映射存在时使用配置权限码。
- 配置映射不存在时继续走自动推断。

### 5. 使用 RemoteServiceName 推断模块名

当前状态：

模块名依赖命名空间硬编码：

```text
.TenantManagement       -> system
.Services.System        -> system
.Services.Monitor       -> monitor
.Services.RecordLog     -> log
.Services.Authentication -> auth
```

固定目标：

- 优先使用显式 `[Permission]` 或配置映射。
- 自动推断模块名时优先从 ABP Conventional Controller 的 `RemoteServiceName` 获取。
- 对当前项目保留兼容映射：

```text
rbac              -> system
tenant-management -> system
audit             -> monitor
default           -> system
```

实现要求：

- `DefaultActionMetadataResolver` 支持从 `ControllerActionDescriptor` 的 endpoint/controller metadata 或 ABP controller setting 中识别 remote service。
- 无法识别时再回退到命名空间。

验收：

- `OperationLogService` 自动推断模块应为 `monitor`，而不是 `system`。
- 新模块接入时，只需配置 remote service 映射，不需要改 namespace 判断。

### 6. 操作日志服务显式标注权限

当前问题：

RBAC 菜单种子已有：

```text
monitor:operlog:list
monitor:operlog:query
monitor:operlog:remove
monitor:operlog:export
```

但 `OperationLogService` 当前未显式声明权限，自动推断可能生成：

```text
system:operation-log:list
system:operation-log:delete
system:operation-log:export
```

固定处理：

在 `AuditLogging.Application.Services.OperationLogService` 方法上加显式权限：

```text
GetListAsync       -> monitor:operlog:list
GetAsync           -> monitor:operlog:query
DeleteAsync        -> monitor:operlog:remove
DeleteCleanAsync   -> monitor:operlog:remove
PostExportAsync    -> monitor:operlog:export
```

同时加：

```text
[OperLogEntity("操作日志")]
```

对写方法加：

```text
[OperLog("删除操作日志", OperEnum.Delete)]
[OperLog("清空操作日志", OperEnum.Clear)]
[OperLog("导出操作日志", OperEnum.Export)]
```

验收：

- 前端操作日志页面按钮权限与后端权限码一致。
- 删除、清空、导出操作产生中文操作日志。

### 7. 清理 OperationOptions 或接入 OperationOptions

当前状态：

`OperationOptions` 存在：

```text
AutoLogWriteOperations
LogReadOperations
IgnoredUrlPrefixes
```

但当前没有实际接入。

固定处理：

接入 `OperationOptions`，不删除。

配置结构：

```json
{
  "Operation": {
    "AutoLogWriteOperations": true,
    "LogReadOperations": false,
    "IgnoredUrlPrefixes": [
      "/api/app/file/",
      "/hangfire"
    ]
  }
}
```

实现要求：

- `DefaultOperationLogMapper` 读取 `IOptions<OperationOptions>`。
- `AutoLogWriteOperations = false` 时，不自动生成写操作日志；只有 `[OperLog]` 显式声明才记录。
- `LogReadOperations = false` 时，查询类操作不记录。
- `IgnoredUrlPrefixes` 命中时不生成 OperationLog。

验收：

- 关闭 `AutoLogWriteOperations` 后普通 `UpdateAsync` 不生成 OperationLog。
- 显式 `[OperLog]` 方法仍生成 OperationLog。
- 忽略 URL 不生成 OperationLog。

## 应补充但不阻塞主链路的事项

### 1. `IgnorePermissionAttribute` 支持类级

当前只支持方法级。

固定处理：

```csharp
[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method)]
```

过滤器同时检查类和方法。

适用场景：

- 某些只读监控 Service。
- 某些公开认证 Service。

### 2. `IgnoreOperLogAttribute` 支持类级

当前只支持方法级。

固定处理：

```csharp
[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method)]
```

Mapper 同时检查类和方法。

适用场景：

- 纯查询 Service。
- 登录日志查询类 Service。
- 文件记录查询类 Service。

### 3. `OperationLogEntry` 与 `IOperationLogStore` 文档口径修正

早期方案写了新增：

```text
OperationLogEntry
IOperationLogStore
```

当前实际实现直接从 `AuditLogInfo` 映射到 `OperationLogEntity`，并通过 `IOperationLogRepository` 保存。

固定处理：

- 不新增 `IOperationLogStore`。
- 不新增 `OperationLogEntry`。
- 文档统一改为：

```text
IOperationLogMapper
OperationLogEntity
IOperationLogRepository
```

原因：

- 现在没有多种日志存储后端，Repository 足够。
- 避免重新引入类似 Security 里的日志持久化抽象。

### 4. 前端 API 路径确认

当前前端请求：

后端由自定义 `YiConventionalRouteBuilder` 生成路由，根路径统一为 `/api`，前端 `VITE_GLOB_API_URL=/api`。

固定验收：

- Swagger 中确认 audit 模块 `OperationLogService` 的实际路由。
- 若后端实际路由不是 `/api/operation-log`，同步修改前端。

## 最终验收清单

### 权限

- 登录、验证码、注册等匿名接口不被权限 Filter 拦截。
- `[AllowAnonymous]` 方法放行。
- `[IgnorePermission]` 方法放行。
- `[Permission]` 显式权限必定校验。
- 自动 CRUD 权限码正确：

```text
GetListAsync         -> {module}:{entity}:list
GetAsync             -> {module}:{entity}:detail
CreateAsync          -> {module}:{entity}:add
UpdateAsync          -> {module}:{entity}:edit
DeleteAsync          -> {module}:{entity}:delete
GetExportExcelAsync  -> {module}:{entity}:export
PostImportExcelAsync -> {module}:{entity}:import
```

- 未解析且无显式权限时，按 `AllowUnresolvedActions` 控制。
- 无权限访问返回 403。
- 有权限访问正常执行业务。

### 操作日志

- `AbpAuditingOptions.IsEnabled = true`。
- `AbpAuditingOptions.IsEnabledForGetRequests = false`。
- `YiAuditLoggingOptions.SaveOperationLog = true`。
- `YiAuditLoggingOptions.SaveAuditLog = false` 时，只保存业务操作日志，不保存详细审计日志。
- `[OperLogEntity("用户")] + UpdateAsync` 生成 `更新用户`。
- `[OperLog("更新用户状态", OperEnum.Update)]` 生成 `更新用户状态`。
- `CreateAsync`、`UpdateAsync`、`DeleteAsync` 自动生成操作日志。
- GET 查询默认不生成 OperationLog。
- 删除、清空、导出操作日志本身能生成操作日志。

### 模块边界

- `Operation.Abstractions` 不包含实体、仓储、SqlSugar、`IAuditingStore`。
- `Operation.Core` 不依赖 RBAC 或 AuditLogging 持久化。
- `AuditLogging.Domain` 不依赖 RBAC。
- `RBAC` 不依赖 `AuditLogging.Domain` 或 `AuditLogging.SqlSugarCore`。
- RBAC 不再包含 `OperationLogEntity`、OperationLog DTO、OperationLogService。

## 实施顺序

按以下顺序执行，避免返工：

1. 修复显式 `[Permission]` 未解析方法不校验的问题。
2. 扩展 `IActionMetadataResolver`，让权限和操作日志共用 `ActionMetadata`。
3. 完整实现 `Post*` / `Put*` / `Delete*` 写操作推断。
4. 增加 `PermissionOptions.Mappings` 并接入权限解析优先级。

7. 接入 `OperationOptions`。
8. 增强 `IgnorePermissionAttribute`、`IgnoreOperLogAttribute` 类级支持。
9. Swagger 确认前端操作日志 API 路径。
10. 最后执行构建和手工链路验证。
