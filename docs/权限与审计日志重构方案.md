# 权限与审计日志重构方案

## 目标

本次重构解决一个核心问题：ABP Conventional Controller 自动生成接口时，无法稳定手写权限校验和操作日志特性。

最终目标：

- 自动生成接口也能按约定获得权限码。
- 自动生成接口也能按约定生成业务操作日志名称。
- 权限和操作日志共享同一套业务 Action 元数据。
- ABP 审计日志能力继续复用 `audit-logging` 模块。
- 横切抽象层不承担实体和持久化职责。

## 设计结论

将当前 `Yi.Framework.Security.*` 改名为更中性的横切业务操作模块：

```text
Yi.Framework.Operation.Abstractions
Yi.Framework.Operation.Core
```

原因：当前模块承载的已不只是安全能力，而是“业务操作”的元数据、权限准入和日志语义。权限和操作日志都围绕同一个 Action/Operation 展开：

```text
UserService.UpdateAsync
  -> 权限：system:user:edit
  -> 日志：更新用户
```

因此叫 `Operation` 比叫 `Security` 更准确。

最终模块职责：

```text
Operation
  负责描述业务操作：权限特性、日志特性、Action 元数据解析、权限准入

AuditLogging
  负责日志留痕：复用 ABP Auditing 捕获请求、异常、耗时、实体变更，并保存审计日志/操作日志

Rbac
  负责权限业务数据：用户、角色、菜单、权限码来源
```

一句话原则：

```text
Operation 识别业务操作并做准入，AuditLogging 负责留痕和日志查询，Rbac 提供权限业务数据。
```

## 当前状态检查

当前 `audit-logging` 模块已基本还原：

- `module/audit-logging/Yi.Framework.AuditLogging.Domain.Shared`
- `module/audit-logging/Yi.Framework.AuditLogging.Domain`
- `module/audit-logging/Yi.Framework.AuditLogging.SqlSugarCore`
- `YiAbpDomainSharedModule` 已依赖 `YiFrameworkAuditLoggingDomainSharedModule`
- `YiAbpDomainModule` 已依赖 `YiFrameworkAuditLoggingDomainModule`
- `YiAbpSqlSugarCoreModule` 已依赖 `YiFrameworkAuditLoggingSqlSugarCoreModule`

仍需收敛的点：

- `Yi.Framework.Security.*` 应改名为 `Yi.Framework.Operation.*`。
- `Security.Core` 仍保留 `OperLogActionFilter`、`DefaultOperLogStore`。
- `Security.Abstractions` 仍保留 `IOperLogStore`、`IOperLogPersistence` 等日志持久化抽象。
- `Rbac.Domain/Persistence/RbacOperLogPersistence.cs` 仍存在。
- `OperationLogEntity`、OperationLog DTO、分页查询服务仍在 RBAC，最终应整体迁入 audit-logging。
- `YiAbpWebModule` 当前同时注册了 `PermissionAuthorizationFilter` 和 `OperLogActionFilter`。
- `AbpAuditingOptions.IsEnabled` 当前被设置为 `false`，这会导致 `audit-logging` 的 `IAuditingStore` 无法发挥作用。

RBAC 旧权限/日志文件已删除但尚未提交时，不恢复原文件。它们的处理原则是：保留删除，迁移有用逻辑。

## 旧 RBAC 文件处理原则

已删除的旧文件不恢复：

- `Domain.Shared/OperLog/OperEnum.cs`
- `Domain.Shared/OperLog/OperLogAttribute.cs`
- `Domain/Authorization/IPermissionHandler.cs`
- `Domain/Authorization/DefaultPermissionHandler.cs`
- `Domain/Authorization/PermissionAttribute.cs`
- `Domain/Authorization/PermissionGlobalAttribute.cs`
- `Domain/Operlog/OperLogGlobalAttribute.cs`

处理方式：

- `OperEnum`、`OperLogAttribute` 迁入 `Operation.Abstractions`。
- `IPermissionHandler`、`PermissionAttribute` 使用 `Operation.Abstractions` 中的新定义。
- `DefaultPermissionHandler` 中有价值的 Claims 判断逻辑迁入 `Operation.Core` 或 `RbacPermissionHandler`。
- `PermissionGlobalAttribute` 中如有白名单、匿名跳过、权限码推断逻辑，只迁移逻辑到 `PermissionAuthorizationFilter` 或 `DefaultActionMetadataResolver`。
- `OperLogGlobalAttribute` 不恢复；如有请求参数、异常、标题推断等逻辑，只作为 `AuditLogging` 的 `OperationLogMapper` 参考。

## 请求链路

运行链路：

```text
HTTP Request
  -> UseAuthentication
  -> PermissionAuthorizationFilter
       -> IActionMetadataResolver
       -> IPermissionCodeGenerator
       -> IPermissionHandler
       -> no permission: 403
  -> Application Service Action
  -> ABP Auditing
       -> AuditLogInfo
       -> Yi.Framework.AuditLogging.Domain.AuditingStore
       -> AuditLogRepository
       -> optional OperationLogMapper
```

权限必须在业务执行前完成，所以保留 `PermissionAuthorizationFilter`。

操作日志不再使用自定义 `OperLogActionFilter`。ABP Auditing 已经能捕获请求、异常、耗时和 Action 信息，操作日志应该作为 ABP 审计信息的一种业务投影。

`audit-logging` 不再做请求过滤，不再判断请求是否允许执行。它只处理 ABP Auditing 生成的 `AuditLogInfo`，并在保存阶段做日志筛选：

```text
请求是否允许执行 -> Operation.PermissionAuthorizationFilter
请求执行后如何留痕 -> ABP Auditing + AuditLogging.AuditingStore
```

日志筛选属于审计结果处理，不属于 MVC 请求过滤：

- 是否保存 GET 请求日志。
- 是否生成 OperationLog。
- 是否忽略某些 URL。
- 是否忽略某些 Service/Method。
- 是否只对写操作生成业务操作日志。

注意：`AbpAuditingOptions.IsEnabled` 是 ABP 审计管道总开关。设置为 `false` 时，ABP 不会生成 `AuditLogInfo`，`AuditingStore.SaveAsync` 不会被调用，因此 `AuditLog` 和 `OperationLog` 都不会记录。

配置原则：

```text
ABP 开关负责“采不采集”。
Yi 开关负责“存不存表”。
```

业务目标固定为“保持审计管道开启，由 audit-logging 控制详细审计日志和业务操作日志是否落库”。因此不关闭 `AbpAuditingOptions.IsEnabled`，使用 audit-logging 自己的保存开关控制落库：

```csharp
Configure<AbpAuditingOptions>(options =>
{
    options.IsEnabled = true;
    options.IsEnabledForGetRequests = false;
});

Configure<YiAuditLoggingOptions>(options =>
{
    options.SaveAuditLog = false;
    options.SaveOperationLog = true;
});
```

分层含义：

```text
AbpAuditingOptions.IsEnabled
  控制 ABP 审计管道是否运行。关闭后 AuditLog 和 OperationLog 都没有。

AbpAuditingOptions.IsEnabledForGetRequests
  控制 GET 请求是否进入 ABP 审计管道。默认 false，查询接口不采集，降低日志量。

YiAuditLoggingOptions.SaveAuditLog
  控制是否保存详细审计日志。

YiAuditLoggingOptions.SaveOperationLog
  控制是否保存业务操作日志。
```

## 权限自动推断规则

默认权限码格式：

```text
{module}:{entity}:{action}
```

示例：

```text
UserService.GetListAsync         -> system:user:list
UserService.GetAsync             -> system:user:detail
UserService.CreateAsync          -> system:user:add
UserService.UpdateAsync          -> system:user:edit
UserService.DeleteAsync          -> system:user:delete
UserService.GetExportExcelAsync  -> system:user:export
UserService.PostImportExcelAsync -> system:user:import
```

推断来源优先级：

1. 显式 `PermissionAttribute`
2. 配置映射表
3. ABP Conventional Controller 的 RemoteServiceName
4. ApplicationService 命名空间和类名
5. 方法名约定

无法解析时：

- 默认放行，降低开源模板和开发期接入成本。
- 由 `PermissionOptions.AllowUnresolvedActions` 控制。
- 生产环境通过 `appsettings.json` 配置为拒绝。

权限准入判断顺序：

```text
1. [AllowAnonymous] -> 放行
2. [IgnorePermission] -> 放行
3. [Permission("...")] -> 校验指定权限码
4. 自动推断出权限码 -> 校验推断权限码
5. 未推断出权限码：
   - AllowUnresolvedActions = true  -> 放行
   - AllowUnresolvedActions = false -> 403
```

`PermissionOptions` 默认值：

```csharp
public class PermissionOptions
{
    public bool AutoCheckResolvedActions { get; set; } = true;
    public bool AllowUnresolvedActions { get; set; } = true;
}
```

`AllowUnresolvedActions = true` 是兼容模式，用于开发期和开源模板快速接入。生产环境通过 `appsettings.json` 关闭：

```json
{
  "Operation": {
    "Permission": {
      "AllowUnresolvedActions": false
    }
  }
}
```

## 操作日志自动推断规则

默认记录写操作：

- `CreateAsync`
- `UpdateAsync`
- `DeleteAsync`
- `ImportAsync`
- `ExportAsync`
- `Post*`
- `Put*`
- `Delete*`

默认不记录查询：

- `GetAsync`
- `GetListAsync`
- `GetSelectDataListAsync`

查询类日志如确实需要，使用配置或 `OperLogAttribute` 显式开启。

### 操作名称生成

操作名称生成规则：

```text
1. 方法上存在 [OperLog]：直接使用特性的 Title 和 OperType。
2. 方法上不存在 [OperLog]：读取 Service 类上的 [OperLogEntity("实体显示名")]。
3. 根据方法名推断操作显示名。
4. 拼接为：{操作显示名}{实体显示名}。
5. 如果缺少实体显示名或无法推断操作类型，则不生成 OperationLog，只保留原始 AuditLog。
```

示例：

```text
[OperLogEntity("用户")] + UpdateAsync -> 更新用户
[OperLogEntity("部门")] + DeleteAsync -> 删除部门
[OperLogEntity("角色")] + CreateAsync -> 添加角色
[OperLog("重置用户密码", OperEnum.Update)] -> 重置用户密码
```

实体显示名放在 Application Service 类上，不放在聚合根上：

```csharp
[OperLogEntity("部门")]
public class DeptService : YiCrudAppService<DeptAggregateRoot, ...>, IDeptService
{
}
```

原因：

- ABP Auditing 捕获的是 `ServiceName` 和 `MethodName`，从 Service 上取元数据最直接。
- 一个聚合根可能被多个 Service 以不同业务视角操作。
- 特殊动作通常是应用层方法语义，不是领域实体语义。
- Application 层不需要为了日志显示名反查 Domain 聚合根，耦合更低。

方法级 `OperLogAttribute` 只用于覆盖特殊动作：

```csharp
[OperLog("更新部门状态", OperEnum.Update)]
public Task UpdateStateAsync(...)
{
}
```

## Web 配置要求

`YiAbpWebModule` 最终应满足：

- 只注册 `PermissionAuthorizationFilter`。
- 不再注册 `OperLogActionFilter`。
- 必须开启 `AbpAuditingOptions.IsEnabled`，否则 AuditLog 和 OperationLog 都不会生成。
- 固定设置 `AbpAuditingOptions.IsEnabledForGetRequests = false`。
- 保留 `AbpAspNetCoreAuditingOptions.IgnoredUrls` 排除文件、Hangfire 等噪音路径。
- 用 `YiAuditLoggingOptions` 控制是否保存详细 AuditLog 和业务 OperationLog。
- 用 `Operation:Permission:AllowUnresolvedActions` 控制未解析接口是否放行。

配置：

```csharp
Configure<AbpAuditingOptions>(options =>
{
    options.IsEnabled = true;
    options.IsEnabledForGetRequests = false;
});
```

日志量控制固定采用非 GET 请求优先：`AbpAuditingOptions.IsEnabledForGetRequests = false`。

`YiAuditLoggingOptions` 默认值：

```csharp
public class YiAuditLoggingOptions
{
    public bool SaveAuditLog { get; set; } = false;
    public bool SaveOperationLog { get; set; } = true;
}
```

`YiAuditLoggingOptions` 绑定到 `appsettings.json`：

```json
{
  "AuditLogging": {
    "SaveAuditLog": false,
    "SaveOperationLog": true
  }
}
```

`YiAbpWebModule` 中绑定配置：

```csharp
Configure<YiAuditLoggingOptions>(
    configuration.GetSection("AuditLogging")
);
```

## 迁移步骤

### Phase 1：完成 audit-logging 还原

1. 确认 audit-logging 三个项目都在解决方案中。
2. 确认 Domain.Shared、Domain、SqlSugarCore 三层依赖已接回主模块。
3. 确认 `IAuditingStore` 实现可被 ABP DI 替换。
4. 构建通过。

### Phase 2：Security 改名为 Operation

1. 将 `Yi.Framework.Security.Abstractions` 改名为 `Yi.Framework.Operation.Abstractions`。
2. 将 `Yi.Framework.Security.Core` 改名为 `Yi.Framework.Operation.Core`。
3. 更新命名空间、项目引用、解决方案引用和模块类名。
4. 保留权限自动推断和权限过滤。
5. 保留操作日志语义特性和枚举。
6. 删除操作日志 Filter 和持久化抽象。

### Phase 3：修正权限准入

1. `PermissionAuthorizationFilter` 改为异步实现。
2. 支持匿名接口跳过。
3. 支持白名单和 `[IgnorePermission]`。
4. 禁止所有未知方法都解析为 `other`。
5. 增加 `AllowUnresolvedActions` 配置，默认值为 `true`；生产配置文件设置为 `false`。
6. 对照已删除的 RBAC 旧权限文件，把仍有价值的 Claims 判断、白名单和跳过逻辑迁入新实现。

### Phase 4：迁移日志语义

1. 在 `Operation.Abstractions` 中保留或新增 `OperLogAttribute`、`OperLogEntityAttribute`、`IgnoreOperLogAttribute`、`OperEnum`。
2. 更新 RBAC 和 Application 中的 using。
3. 为核心 Application Service 补充类级 `[OperLogEntity("...")]`。
4. 将 OperationLog DTO、实体、查询服务整体迁入 audit-logging，不做短期保留。
5. 对照已删除的 `OperLogGlobalAttribute`，只迁移必要的日志字段采集和标题推断逻辑，不恢复旧 Filter。

### Phase 5：复用 ABP Auditing 生成操作日志

1. 在 audit-logging 中增加 `OperationLogEntry` 和映射器。
2. 从 `AuditLogInfo.Actions` 找到业务 Action。
3. 结合 Operation 元数据生成标题、操作类型、是否成功、耗时、请求路径、异常信息。
4. 保存到 OperationLog。
5. 新增 `YiAuditLoggingOptions`，区分 `SaveAuditLog` 和 `SaveOperationLog`。
6. 在 audit-logging 中提供 OperationLog 分页查询、删除、清空、导出等应用服务。
7. Web 动态 API 注册 audit-logging Application 模块，前端操作日志页面改为调用 audit-logging API。

### Phase 6：验证

必须验证：

- 登录、验证码、注册等匿名接口不被权限 Filter 拦截。
- 自动生成 CRUD 接口能推断出正确权限码。
- 无权限访问返回 403。
- 有权限访问正常执行业务。
- 写操作产生 AuditLog。
- 写操作产生 OperationLog。
- GET 查询默认不产生 OperationLog。
- `dotnet build Yi.Abp/Yi.Abp.slnx` 通过。
